Architecture Overview

TXR Automation follows a four-layer architecture with clear separation of concerns. The core library provides shared functionality consumed by all processing modules, which are exposed through CLI scripts and a desktop GUI.

Design Philosophy: Minimise external dependencies, maximise code reuse through a shared core library, use dataclasses for all data transfer, CSV-first I/O (no Excel), and singleton reference data managers for performance.

Layer Responsibilities

Layer Responsibility Key Principle
User Interface CLI argument parsing, GUI event handling, user interaction Thin layer — delegates all logic downward
Processing Business logic, validation rules, data transformation Domain-specific logic, uses core library for common operations
Core Library Shared configuration, logging, reference data, validation utilities Single source of truth for common functionality
Data Persistent storage, file I/O, caching CSV-first for I/O, SQLite for caching, YAML for config

Module Architecture

Core Module (src/core/)

The foundation library providing shared functionality across all packages. Version: 1.1.0.

Subpackage Key Classes Purpose
config/ ConfigManager, PathConfig, ProcessorConfig YAML loading with LRU caching, environment variable override, priority cascade
data/ CountryDataManager, IDFormatManager, INCIDENT_CODE_MATRIX ISO 3166-1 country codes (EEA classification), 68 ID regex patterns, incident metadata
logging/ StructuredLogger, create_logger() JSON structured logging with file + console output, statistics protocol
utils/ DateParser, FileDiscovery, safe_open_csv Multi-format date parsing with caching, file discovery, safe CSV operations
validation/ validate_id, validate_id_auto, validate_not_empty ID format validation against regex patterns, common validation utilities

Accuracy Testing Module (src/accuracy_testing/)

The largest module, containing all accuracy testing validation logic. Version: 3.0.0.

Component Key Classes Purpose
processor.py
(2,611 LOC)		 IDValidationProcessor, InconsistentIDProcessor, ClientRecord, AccuracyConfigManager Main validation pipeline: format check → logic check → correction → CONCAT generation. Also handles inconsistent ID grouping by Person Code.
id_logic_validator.py
(1,125 LOC)		 IDLogicValidator Validates embedded information in national IDs: DOB extraction, gender codes, check digits (Modulus 97, Modulus 11, Luhn). Covers 15 countries.
models/ DecisionMakerRecord, PricingRecord, NetAmountRecord, NetQuantityRecord, DataPushRecord Dataclass models with factory methods (from_dict(), from_row()) for each validation type
validators/ DecisionMakerValidator, PricingValidator, NetAmountValidator, NetQuantityValidator, DataPushProcessor Dedicated validator classes for each incident type, with batch processing support
scripts/ 15 CLI entry points Thin CLI wrappers: argument parsing → config loading → processor invocation → output writing
sql_templates/ 11 SQL + 1 DTF template Parameterised SQL query templates with transaction reference batching (groups of 900)

FIRDS Module (src/firds/)

Self-contained module for FCA FIRDS instrument reportability.

Component Purpose
FirdsApiClient Queries FCA Data API for FULINS/DLTINS/FULCAN file listings
FirdsDownloader Downloads and extracts XML files (8 MB chunk size)
FirdsXmlParser Memory-efficient streaming XML parser using iterparse
FirdsCacheManager SQLite database: instruments table with upsert, termination, cancellation; firds_sync_log for idempotency
FirdsRefresher Orchestrates full weekly rebuilds and daily delta refreshes (5,000-record upsert batches)
FirdsReportabilityChecker Determines reportability: ACTIVE, NOT_IN_FIRDS, ADMISSION_AFTER_TRADE, TERMINATED_BEFORE_TRADE, CANCELLED

GLEIF Module (src/gleif/)

Self-contained module for GLEIF LEI validation and entity lookup.

Component Purpose
GleifApiClient Queries GLEIF API v1 for LEI records, ISIN mappings, BIC lookups (rate limit: 60/min)
GleifDownloader Downloads Golden Copy files (300s timeout, custom User-Agent)
GleifCsvParser Streaming parser for 3.2M-record Golden Copy CSV, row-by-row processing
GleifCacheManager SQLite: lei_records, lei_isin_map, lei_fts (FTS5 full-text search), gleif_sync_log
GleifRefresher Full rebuild + delta refresh cycles (8h, 24h, 7d, 31d) with 5,000-record batches
GleifLookup LEI validation with registration status, trade-date awareness, name search, BIC lookup

Replay Module (src/replay/)

Script Version Purpose
phase_2_processor.py v4.2 Transaction reference matching with hash table indexing (O(1) lookups), character encoding handling
phase_3_processor.py v5.2 Client record matching: first name, surname, DOB, ID value. Fuzzy matching, correction-bearing row preference
phase_3_final_lookup.py — UnaVista transaction validation — verifies proposed corrections against source data
merge_inconsistent_ids.py — Merges duplicate rows in inconsistent ID summary files

Design Patterns

Pattern Implementation Where Used
Singleton Single instance of reference data loaded once, cached globally country_manager, id_format_manager (lazy-loaded at import time)
Factory Factory functions create instances from configuration create_logger(), ClientRecord.from_dict(), DecisionMakerRecord.from_row()
Strategy Different validation strategies selected by country code or ID type IDLogicValidator dispatches to 15 country-specific validators
Template Method Common processing pipeline shared across scripts All CLI scripts: parse args → load config → read CSV → process → write CSV → log summary
Configuration Cascade Layered config resolution: env vars → YAML → defaults ConfigManager, AccuracyConfigManager
Repository / Cache SQLite databases with upsert semantics and sync logs FirdsCacheManager, GleifCacheManager
Observer Structured logging with statistics collection StructuredLogger with StatsProtocol
Dataclass / DTO Immutable data transfer objects with factory methods ClientRecord, DecisionMakerRecord, PricingRecord, ReportabilityResult, etc.
Worker Thread QThread background execution with signal/slot communication ScriptRunnerWorker in GUI module

Key Dependency Rules

  • Core depends on nothing — it is the foundation layer
  • Processing modules depend on Core — for config, logging, validation, reference data
  • FIRDS and GLEIF are independent — they use Core for logging but are otherwise self-contained
  • GUI depends on all processing modules — it invokes them as CLI scripts via QThread workers
  • No circular dependencies — dependency flow is strictly top-down
