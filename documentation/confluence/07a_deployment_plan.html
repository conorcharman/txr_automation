Phase 7: Deployment & Transition

Objective

Deploy the Python toolset to production with a parallel-running period and managed VBA retirement.

Deployment Steps

StepDescriptionDurationOwner
7.1 Environment provisioning Install Conda environment on production workstations; verify pip install -e .; confirm all 22 console scripts are accessible 1–2 days Developer
7.2 Parallel running setup Run both VBA macros and Python scripts on the same quarterly datasets for one full quarter; capture outputs side-by-side 1 quarter Developer + Analysts
7.3 Output comparison Diff VBA and Python outputs column-by-column; document any discrepancies; resolve differences 1–2 weeks Developer
7.4 User training Conduct training sessions covering CLI usage, configuration, troubleshooting; distribute user guides 1 week Developer + Team Lead
7.5 VBA retirement Archive VBA macros (already in legacy/vba/); remove VBA dependencies from analyst workbooks; update team procedures 1–2 days Developer
7.6 Production cutover Designate Python as the primary toolset; VBA retained as read-only reference only 1 day Team Lead

Acceptance Criteria

  • Python outputs match VBA outputs for one full quarterly dataset (zero unexplained discrepancies)
  • All analysts can independently run the validation workflow end-to-end
  • All 466 tests pass in the production environment
  • Performance benchmarks meet or exceed targets (see Performance Benchmarks)

Phase 8: Unified CLI Tool

Detailed plan at documentation/planning/Phase_8_CLI_Tool_Plan.md.

Concept

Consolidate all 22 console scripts into a single interactive CLI entry point (txr-automation) with text-based menus:

$ txr-automation

╔══════════════════════════════════════════════════╗
║     Transaction Reporting Automation Tool         ║
╚══════════════════════════════════════════════════╝

What would you like to do?

  [1] Extract Generation
  [2] Accuracy Testing
  [3] Replay Processing
  [4] View Recent Logs
  [5] Configuration
  [Q] Quit

Key Design Decisions

  • Single entry point: txr-automation replaces memorising 22 separate commands
  • Guided workflow: Interactive prompts for incident code selection and configuration
  • Business logic separation: CLI layer calls existing main() functions; no duplication
  • GUI foundation: Architecture designed to support the PySide6 GUI as an equivalent front-end

Estimated Effort

3–4 weeks development + 1 week testing. Depends on Phase 7 completion.

Installation

# Clone repository
git clone <repository-url> txr_automation
cd txr_automation

# Create and activate Conda environment
conda env create -f environment.yml
conda activate txr_automation

# Install in editable mode
pip install -e .

# Verify installation
validate-all --help
pytest tests/ -v

Configuration

# Copy configuration templates
cp config/templates/*.yaml config/local/

Risks & Mitigations

RiskImpactLikelihoodMitigation
VBA/Python output discrepancies during parallel running High Low 466 tests provide high confidence; column-by-column diff tooling prepared
Analysts unable to adopt CLI workflow Medium Medium GUI application provides non-CLI alternative; training sessions planned
Conda environment conflicts on production machines Medium Low environment.yml pins all dependencies; isolated Conda environment
Performance degradation with very large datasets Low Low Benchmarked to 1.5M records (11K records/second); typical workloads are <50K records
FIRDS/GLEIF API availability Medium Low Local SQLite caches provide offline operation after initial download
