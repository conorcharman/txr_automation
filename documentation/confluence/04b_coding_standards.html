Import Order

Imports follow a four-group convention, separated by blank lines:

# 1. Standard library
import os
import sys
from pathlib import Path
from typing import List, Optional, Dict

# 2. Third-party packages
import pandas as pd
import click

# 3. Core library imports
from core import (
    StructuredLogger,
    create_logger,
    country_manager,
    id_format_manager,
)

# 4. Local package imports
from .processor import ClientRecord, IDValidationProcessor
Rule: Never use wildcard imports (from module import *). Always import specific names.

Docstrings (Google Style)

def validate_id_format(
    country: str,
    id_type: str,
    id_value: str,
) -> bool:
    """
    Validate an ID value against the expected format for a country/type pair.

    Checks the ID value against compiled regex patterns from the ESMA CID
    guidance. Returns True if the value matches the expected format.

    Args:
        country: ISO 3166-1 alpha-2 country code (e.g., "GB", "DE")
        id_type: ID type code (e.g., "NIDN", "CCPT", "CONCAT")
        id_value: The ID value to validate

    Returns:
        True if the ID value matches the expected format, False otherwise

    Raises:
        ValueError: If country or id_type is not recognised

    Example:
        >>> validate_id_format("GB", "NIDN", "AB123456C")
        True
        >>> validate_id_format("GB", "NIDN", "INVALID")
        False
    """

Required on: all public functions, classes, class methods, and module-level docstrings. Private methods (_name) should have at least a one-line docstring.

Dataclasses

All data transfer objects (DTOs) use Python @dataclass:

from dataclasses import dataclass, field
from typing import Optional, List

@dataclass
class ClientRecord:
    """Represents a client record for ID validation."""

    transaction_ref: str
    account_id: str
    person_code: str
    id_value: str = ""
    id_type: str = ""
    correction: str = ""
    error_flag: str = "N"
    error_details: List[str] = field(default_factory=list)

    @classmethod
    def from_dict(cls, data: dict) -> "ClientRecord":
        """Create a ClientRecord from a dictionary (e.g., CSV row)."""
        return cls(
            transaction_ref=data.get("Transaction_Ref", ""),
            account_id=data.get("Account_ID", ""),
            person_code=data.get("Person_Code", ""),
        )

Conventions

  • Use @dataclass (not NamedTuple) for mutable records
  • Always provide a from_dict() or from_row() factory classmethod for CSV/dict construction
  • Use field(default_factory=list) for mutable defaults
  • Keep field names consistent with CSV column names where possible

Error Handling

try:
    result = process_record(record)
except ValueError as e:
    logger.warning(
        f"Validation failed for {record.transaction_ref}: {e}"
    )
    record.error_flag = "Y"
except Exception as e:
    logger.error(
        f"Unexpected error processing {record.transaction_ref}: {e}"
    )
    raise

Rules

  • Catch specific exceptions — never bare except:
  • Log before re-raising — ensures context is captured in structured logs
  • Validation errors are warnings — they set error_flag = "Y" but do not stop processing
  • System errors are fatal — file I/O failures, config errors, etc. should raise and halt

Logging

Use the structured logging framework from core:

from core import create_logger

logger = create_logger(__name__)

# Informational
logger.info("Processing batch", extra={
    "batch_size": len(records),
    "incident_code": "7_39",
})

# Warning (validation failure)
logger.warning("Invalid ID format", extra={
    "person_code": record.person_code,
    "expected": "AB123456C",
    "actual": record.id_value,
})

# Error (system failure)
logger.error("Failed to read input file", extra={
    "path": str(input_path),
    "error": str(e),
})
Never log sensitive data. Do not log full ID values, names, or account numbers in production. Use truncated or hashed values for debugging.

VBA-to-Python Conversion Patterns

When migrating VBA functions to Python, use these standard mappings:

VBA Pattern Python Equivalent
Dim x As String x: str = ""
For i = 1 To n for i in range(1, n + 1):
Left(str, n) str[:n]
Right(str, n) str[-n:]
Mid(str, start, len) str[start-1:start-1+len]
Len(str) len(str)
UCase(str) / LCase(str) str.upper() / str.lower()
Trim(str) str.strip()
IsNumeric(x) str(x).isdigit() or regex
InStr(str, sub) sub in str or str.find(sub)
Split(str, delim) str.split(delim)
Join(arr, delim) delim.join(arr)
Range("A1").Value pandas DataFrame or CSV reader
Cells(row, col).Value df.iloc[row, col]

Migration Checklist

  1. Read and understand the VBA source completely
  2. Document VBA-to-Python function mapping
  3. Create test fixtures from VBA expected outputs
  4. Write unit tests for validation logic
  5. Implement core validation using existing utilities
  6. Create CLI script following established patterns
  7. Add integration tests
  8. Verify output matches VBA exactly
  9. Update documentation and exports

Common Pitfalls to Avoid

  1. Don’t reinvent utilities — check src/core/ and src/accuracy_testing/core/ before writing new helper functions
  2. Don’t use Excel libraries — all I/O is CSV-based. Use pandas or csv module
  3. Don’t hardcode paths — all paths come from YAML configuration or environment variables
  4. Don’t skip type hints — required on every function and method
  5. Don’t use American English in documentation
  6. Don’t skip tests — every function needs unit tests before merging
  7. Don’t modify VBA files in legacy/vba/ — they are read-only reference
  8. Don’t create duplicate reference data — use existing CSVs in documentation/reference_data/
