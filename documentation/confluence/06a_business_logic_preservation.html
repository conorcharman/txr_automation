Preservation Approach

The migration followed a strict behaviour-equivalent methodology:

  1. Read: Fully understand each VBA function’s logic before writing Python
  2. Map: Document the VBA function → Python function correspondence (see Traceability Matrix)
  3. Implement: Preserve identical decision paths, including edge-case handling
  4. Test: Write unit tests using expected outputs derived from VBA execution
  5. Verify: Run both VBA and Python on identical input data and compare results

Incident Code Routing

The INCIDENT_CODE_MATRIX in core/data/incident_codes.py is the single source of truth for routing transactions to the correct validation type. Each entry specifies:

  • sides — {'buyer'}, {'seller'}, {'buyer', 'seller'}, or set() (trade-level)
  • validation_type — routing key: standard_id, decision_maker, inconsistent_id, standard_name, non_zero_net_qty, non_zero_net_amt, pricing, or pending
  • description — human-readable rule description

Verification Methodology

Automated Test Coverage

  • 466 unit and integration tests — all passing (100%)
  • Tests derived from VBA expected outputs using known valid/invalid IDs
  • Edge cases: empty values, special characters, boundary dates, century transitions
  • Parametrised tests cover multiple countries per test function

Manual Verification Steps Performed

  1. Side-by-side execution: Both VBA and Python run on identical sample data; outputs compared column-by-column
  2. Check digit verification: Known valid NIDNs from national registries verified against both VBA and Python check digit implementations
  3. CONCAT generation parity: 100+ CONCAT values generated by VBA matched against Python output
  4. Correction logic parity: VBA-generated corrections compared against Python corrections for same input records
  5. Incident code routing: All active incident codes routed through both VBA and Python pipelines to confirm identical script selection
