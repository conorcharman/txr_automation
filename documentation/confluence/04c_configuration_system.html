Overview

TXR Automation uses a layered YAML configuration system with two independently developed but compatible managers:

ManagerModulePurpose
ConfigManager src/core/config/config_manager.py Shared core config for replay processing. Provides PathConfig and ProcessorConfig dataclasses.
AccuracyConfigManager src/accuracy_testing/processor.py Accuracy testing config with batch/single mode support. Provides AccuracyPathConfig and AccuracyProcessorConfig dataclasses.

Both managers follow the same design principles: YAML files as the primary source, environment variable overrides with TXR_ prefix, and dataclass-based typed configuration objects.

Architecture

┌──────────────────────────────────────────────────────────┐
│                  Configuration Sources                    │
├──────────────────────────────────────────────────────────┤
│  CLI Arguments       (--config, --log-level, etc.)       │  ← Highest priority
│  Environment Vars    (TXR_ACCURACY_PATHS_INPUT_FILE)     │
│  YAML Config File    (config/local/.../*.yaml)           │
│  Default Values      (in dataclass field defaults)       │  ← Lowest priority
└──────────────────────────────────────────────────────────┘
                          │
                          ▼
┌──────────────────────────────────────────────────────────┐
│              Config Manager (load + merge)                │
├──────────────────────────────────────────────────────────┤
│  1. Load YAML file → dict                                │
│  2. Load env vars  → dict                                │
│  3. Merge (env overrides YAML)                           │
│  4. Parse into typed dataclass                           │
└──────────────────────────────────────────────────────────┘
                          │
                          ▼
┌──────────────────────────────────────────────────────────┐
│         Typed Config Objects (dataclasses)                │
├──────────────────────────────────────────────────────────┤
│  AccuracyPathConfig        AccuracyProcessorConfig       │
│  PathConfig (replay)       ProcessorConfig (replay)      │
└──────────────────────────────────────────────────────────┘

Creating Local Configs

# Create the local config directory structure
mkdir -p config/local/accuracy_testing

# Copy a template
cp config/templates/accuracy_testing/buyer_validation_template.yaml \
   config/local/accuracy_testing/buyer_validation.yaml

# Edit to set your local file paths
$EDITOR config/local/accuracy_testing/buyer_validation.yaml

Single Mode Configuration

Use mode: "single" to process one incident at a time with explicit file paths:

mode: "single"

testing_period:
  fiscal_year: "FY26"
  quarter: "Q1"

single:
  incident_code: "7_37"
  
  paths:
    # Input: SQL extract from database
    input_file: "data/extracts/7_37_FY26_Q1.csv"
    
    # Reference: Kaizen template with expected values
    template_file: "data/templates/FY26 Q1 7_37.csv"
    
    # Template column mapping
    template_id_column: "Buyer identification code"
    template_type_column: "Type of buyer identification code"
    
    # Output: validation results
    output_file: "data/validated/validated_FY26_Q1_7_37.csv"
    
    # Optional tracker files
    # italian_tracker: "data/trackers/italian_tracker.csv"
    # main_tracker: "data/trackers/main_tracker.csv"
    
    # Log directory
    log_output: "logs"

Batch Mode Configuration

Use mode: "batch" to process multiple incidents automatically. The system constructs file paths using patterns and auto-discovers incidents:

mode: "batch"

testing_period:
  fiscal_year: "FY26"
  quarter: "Q1"

batch:
  # "auto" discovers all incidents, or list specific ones: ["7_35", "7_37"]
  incidents: "auto"
  
  # Auto-discovery incident list
  auto_incidents:
    - "7_35"    # Incorrect CONCAT value
    - "7_37"    # Incorrect NIDN value
    - "7_39"    # Incorrect CCPT value
  
  paths:
    extract_dir: "data/extracts/FY26/Q1"
    template_dir: "data/templates/FY26/Q1"
    output_dir: "data/validated/FY26/Q1"
    log_output: "logs"
  
  # Filename pattern construction
  # Variables: {incident}, {fiscal_year}, {quarter}
  filename_patterns:
    extract: "{incident}_{fiscal_year}_{quarter}.csv"
    template: "{fiscal_year} {quarter} {incident}.csv"
    output: "validated_{fiscal_year}_{quarter}_{incident}.csv"
How auto-discovery works: When incidents: "auto" is set, the processor scans extract_dir for files matching the filename_patterns.extract pattern for each incident in auto_incidents. Only incidents with matching extract files are processed.

Usage with --use-env Flag

# Set environment variables
export TXR_ACCURACY_PATHS_INPUT_FILE="data/extracts/7_37_FY26_Q1.csv"
export TXR_ACCURACY_PATHS_OUTPUT_FILE="data/validated/validated_7_37.csv"

# Run using environment variables instead of a YAML file
buyer-id-validation --use-env

Replay Configuration

Replay processing uses the core ConfigManager with a simpler flat structure:

# config/local/replay/phase2.yaml
paths:
  replay_input: "data/replay/input"
  incident_files: "data/replay/incidents"
  replay_output: "data/replay/output"
  log_output: "logs"

processing:
  batch_size: 50
  log_level: "INFO"
  enable_progress_reporting: true
  encoding: "utf-8"
