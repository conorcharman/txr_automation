ID Validation Pipeline

The buyer/seller ID validation is the most complex flow, processing records through four phases:

 Input CSV (Template with transaction data)
 |
 v
 +-------------------------------------------------------+
 |  PHASE 1: Load & Parse                                 |
 |  * Read CSV into ClientRecord dataclasses              |
 |  * Determine client_type (buyer/seller)                |
 |  * Load reference data (singletons, cached)            |
 +--------------------------+----------------------------+
                            |
                            v
 +-------------------------------------------------------+
 |  PHASE 2: Format Validation                            |
 |  * Extract priority country from nationality           |
 |    (EEA countries prioritised over ROW)                |
 |  * Validate ID value against country-specific regex    |
 |    (68 patterns across 27 countries)                   |
 |  * Test alternative ID types if primary fails          |
 |  * Handle JNT (joint) account aggregation              |
 |  * Apply Swedish century logic for SE NIDN             |
 +--------------------------+----------------------------+
                            |
                            v
 +-------------------------------------------------------+
 |  PHASE 3: Logic Validation                             |
 |  * Check embedded DOB matches provided DOB             |
 |  * Check embedded gender code matches provided         |
 |  * Run check digit / checksum algorithms:              |
 |    - Modulus 97 (BE, SI)                               |
 |    - Modulus 11 (DK, EE, IS, LT, LV, NO)              |
 |    - Luhn (SE)                                         |
 |    - Weighted checksums (FI, IT, SK)                   |
 |  * Italian tracker lookup (confirmed fiscal codes)     |
 +--------------------------+----------------------------+
                            |
                            v
 +-------------------------------------------------------+
 |  PHASE 4: Correction & Output                          |
 |  * Generate CONCAT ID if no valid format found         |
 |    (CONCAT = country + cleaned first name + surname    |
 |     + DOB in YYYYMMDD format)                          |
 |  * Perform template validation against master file     |
 |  * Set error flags and correction columns              |
 |  * Write main output CSV + errors-only CSV             |
 +-------------------------------------------------------+

Configuration Cascade

Configuration values are resolved in priority order (highest wins):

 Priority 1: Environment Variables (TXR_*)
        v (if not set)
 Priority 2: YAML Configuration File
        v (if not specified)
 Priority 3: Hardcoded Defaults
Source Example When Used
Environment variable TXR_LOG_LEVEL=DEBUG CI/CD pipelines, temporary overrides, secrets
YAML file --config config/local/buyer_id.yaml Normal operation, per-user customisation
Default value log_level = "INFO" Development, quick testing

Accuracy Testing: Batch vs Single Mode

Mode Config Key Behaviour
Batch mode: "batch" Auto-discovers files matching filename patterns. Processes multiple incidents. Uses incidents: "auto" or "all" or explicit list.
Single mode: "single" Explicit file paths provided. Processes one incident at a time.

Reference Data Loading

Reference data is loaded once at module import time and cached globally via singleton managers:

Manager Data Source Records Access Pattern
country_manager Embedded in core/data/country_codes.py ISO 3166-1 countries with EEA classification country_manager.is_eea_country("GB")
id_format_manager Embedded in core/data/id_formats.py 68 regex patterns for NIDN/CCPT/CONCAT across 27 countries id_format_manager.validate_format("GB", "NIDN", value)
INCIDENT_CODE_MATRIX Embedded in core/data/incident_codes.py Incident metadata: type, client side, validation category is_buyer_incident("7_39")

Additionally, the IDLogicValidator embeds country-specific constants used for validation:

  • IT_MONTH_LETTERS — Italian fiscal code month-letter mapping
  • FI_CENTURY_MARKERS — Finnish century separator characters
  • LT_CENTURY_GENDER_CODES — Lithuanian century/gender code table
  • LV_CENTURY_CODES — Latvian century code mapping

Scheduler Pipeline Flow

The desktop GUI scheduler engine executes multi-step pipelines automatically on a cron schedule:

 Schedule Trigger (cron expression via QSettings)
 |
 v
 +------------------------------------------------+
 |  STEP 1: EXTRACT                                |
 |  * generate-sql-extract / generate-period-extract|
 |  * Produce .sql/.dtf files for DB execution      |
 +---------------------+--------------------------+
                       |
                       v
 +------------------------------------------------+
 |  STEP 2: COLLATE                                |
 |  * collate-csv-extracts                         |
 |  * Merge DB extract CSVs into unified files     |
 +---------------------+--------------------------+
                       |
                       v
 +------------------------------------------------+
 |  STEP 3: VALIDATE                               |
 |  * validate-all (runs all applicable incidents) |
 |  * Produces validated CSV + errors-only CSV     |
 +---------------------+--------------------------+
                       |
                       v
 +------------------------------------------------+
 |  STEP 4: PUSH                                   |
 |  * data-push                                    |
 |  * Update master tracking files                 |
 +------------------------------------------------+

  Pipeline output files use date-stamped naming.
  Run history and status tracked in SchedulerTab.
  Failures trigger Windows toast notifications via TrayApp.
