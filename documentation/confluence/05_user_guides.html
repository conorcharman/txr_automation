Accuracy Testing

The accuracy testing suite validates transaction data against regulatory requirements for UK MiFIR/RTS 22 compliance. It covers 11 incident types across buyer/seller identification, decision maker validation, and pricing checks.

Workflow Overview

  1. Extract data — generate SQL queries to extract data from the source database
  2. Run validation — validate extracted data against format rules, logic checks, and Kaizen templates
  3. Review results — examine errors-only CSV and full results CSV
  4. Push corrections — push validated corrections to the master tracking file

Step 1: Generate SQL Extracts

# Generate SQL queries for buyer ID incidents (batch mode)
sql-extract --config config/local/accuracy_testing/sql_extract.yaml

# Or for a specific incident
sql-extract --config config/local/accuracy_testing/sql_extract.yaml --incident 7_37

SQL output files are written to the configured output directory, ready to run against the source database.

Step 2: Buyer/Seller ID Validation

Incidents covered: 7_35, 7_37, 7_39 (buyer), 16_19, 16_21, 16_23 (seller) — CONCAT, NIDN, and CCPT ID types. 
# Single incident
buyer-id-validation --config config/local/accuracy_testing/buyer_validation.yaml

# Batch mode (processes all auto-discovered incidents)
# Set mode: "batch" in the YAML config file
buyer-id-validation --config config/local/accuracy_testing/buyer_validation.yaml

# Dry run (preview without writing output)
buyer-id-validation --config config/local/accuracy_testing/buyer_validation.yaml --dry-run

# Seller side (identical workflow)
seller-id-validation --config config/local/accuracy_testing/seller_validation.yaml

What the Validation Does

  1. Loads extract CSV and (optionally) Kaizen template CSV
  2. For each record:
    • Determines priority country from nationality (EEA first, then ROW, alphabetical within group)
    • Validates ID format against 68 country-specific regex patterns
    • If NIDN: runs embedded logic validation (DOB extraction, checksums) for 15 countries
    • Tests alternative ID types if primary fails
    • Generates CONCAT if ID is invalid (country code + first name + surname + DOB)
    • Looks up record in Kaizen template for expected value comparison
  3. Writes full results CSV + errors-only CSV + log file

Output Columns Added

Column Description
Error Y/N flag — Y if any validation failed
Error_Details Human-readable error description
Correction Suggested correction value (if applicable)
Match Template match result (MATCH / MISMATCH / NOT_FOUND)
Expected_ID Expected ID from Kaizen template
Expected_Type Expected ID type from Kaizen template

Step 3: Inconsistent ID Validation

Incidents covered: 7_66 (buyer), 16_20 (seller) — detects when the same person has different IDs over time. 
inconsistent-buyer-id-validation \
    --config config/local/accuracy_testing/inconsistent_buyer_validation.yaml

inconsistent-seller-id-validation \
    --config config/local/accuracy_testing/inconsistent_seller_validation.yaml

Algorithm

  1. Groups records by Person Code
  2. Sorts each group chronologically by Trade_Date_Time
  3. Validates each ID (format + logic)
  4. Detects inconsistencies using prefix-aware nationality grouping
  5. Corrects INVALID IDs using the most recent PRIOR VALID ID in the same group
  6. Valid-to-valid ID changes are preserved (not flagged as errors)

Step 4: Decision Maker Validation

validate-ftbdm --config config/local/accuracy_testing/ftbdm_validation.yaml
validate-ftsdm --config config/local/accuracy_testing/ftsdm_validation.yaml

Validates Field 27 (buyer DM) and Field 28 (seller DM) decision maker codes. For Discretionary (D) service level accounts, the DM code must exist and differ from the party code. LEI format validation is performed on DM codes.

Step 5: Pricing Validation

pricing-validation --config config/local/accuracy_testing/pricing_validation.yaml

Verifies that Net Amount = Consideration + Interest for each transaction record. Flags discrepancies beyond a configurable tolerance threshold.

Step 6: Push Results

data-push --config config/local/accuracy_testing/data_push.yaml

Pushes validated corrections from the output CSV to the master tracker file by matching on transaction reference.

Step 7: Run All Validations

# Orchestrate all validation scripts in sequence
run-all-validations --config config/local/accuracy_testing/run_all_validations.yaml

Replay Processing

Replay processing matches transaction records against source data to identify and correct reporting errors. It operates in three phases.

Phase 2: Transaction Reference Matching

replay-phase2 --config config/local/replay/phase2.yaml

Matches replay records to source data using transaction reference numbers. Uses hash table indexing for O(1) lookups. Handles character encoding issues from AS/400 exports.

Phase 3: Client Record Matching

replay-phase3 --config config/local/replay/phase3.yaml

Matches client records by first name, surname, date of birth, and ID value. Supports fuzzy matching with configurable thresholds. Prefers correction-bearing rows when multiple matches exist.

Phase 3 Final Lookup

replay-phase3-final --config config/local/replay/phase3_final.yaml

Verifies proposed corrections from Phase 3 against UnaVista source data.

XLSX to CSV Converter

replay-xlsx-converter --input data/replay/input.xlsx --output data/replay/input.csv

Converts XLSX files to CSV format for pipeline input.

FIRDS Reportability

The FIRDS module maintains a local SQLite cache of FCA Financial Instruments Reference Data System (FIRDS) data for determining whether instruments are MiFIR reportable.

Refresh the Cache

# Full refresh — downloads and rebuilds the entire cache
firds-refresh --type full --config config/local/firds_config.yaml

# Delta refresh — updates only new/changed records
firds-refresh --type delta --config config/local/firds_config.yaml

Check Reportability

# Check a single ISIN
firds-check --isin GB0007188757

# Batch check from CSV
firds-check --input data/isins_to_check.csv --output data/reportability_results.csv

GLEIF LEI Lookup

The GLEIF module maintains a local SQLite cache (with FTS5 full-text search) of the GLEIF Legal Entity Identifier golden copy for LEI validation and entity lookups.

Refresh the Cache

# Full refresh — downloads GLEIF golden copy and rebuilds cache
gleif-refresh --type full --config config/local/gleif_config.yaml

# Backfill missing LEIs from API
gleif-backfill --config config/local/gleif_config.yaml

Look Up an LEI

# Single LEI lookup
gleif-check --lei 529900T8BM49AURSDO55

# Search by entity name (uses FTS5 full-text search)
gleif-check --name "Example Corporation"

# Batch lookup from CSV
gleif-check --input data/leis_to_check.csv --output data/lei_results.csv

System Tray Application

# Launch the tray service
txr-tray

The tray app runs schedules in the background with Windows toast notifications. It can be configured to auto-start on login.

Note: The GUI is in active development. All validation logic is shared with the CLI scripts, so results are identical regardless of interface used.

Web Application

The fullstack web application provides a browser-based interface to all TXR Automation functionality, deployed via Docker Compose. It comprises a React 19 single-page application (SPA) served by Nginx, a FastAPI backend, Celery task workers, and PostgreSQL/Redis persistence.

Starting the Web Application

# Start all services (development — source-mounted, hot reload)
docker compose up --build

# Production (optimised images, health checks, memory limits)
docker compose -f docker-compose.prod.yml up --build

# Access the web interface at http://localhost:3000
# API documentation (dev only) at http://localhost:8000/docs

Pages

The SPA uses React Router and provides the following 10 pages:

Saved Configurations

All script forms in the web application support saving and reloading parameter sets via the ConfigLoader component. Configurations are stored server-side in PostgreSQL and namespaced by script name, making them available across browser sessions and machines.

# REST endpoints used by ConfigLoader
GET    /api/configs?script_name=buyer_id_validation  — list saved configs for a script
POST   /api/configs                                   — save current form values
PUT    /api/configs/{id}                              — update a saved config
DELETE /api/configs/{id}                              — remove a saved config

Docker Services

The web application comprises 6 Docker services:

  • Web (Nginx) — serves the React SPA on port 3000 (dev) / 80 (prod) and reverse-proxies /api/ws/ (WebSocket, 86400 s timeout) and /api/ (REST) to the API service; all unmatched routes fall back to index.html for client-side routing
  • API (FastAPI/Uvicorn) — REST API with 13 route groups and an OpenAPI schema available in dev at /docs (disabled in production)
  • Celery Worker — executes validation scripts asynchronously; captures stdout/stderr and publishes log lines to Redis pub/sub
  • Celery Beat — periodic task scheduler; fires check_and_run_schedules to trigger due schedules, pipelines, and reconciliation runs
  • Redis 7 — message broker for Celery and pub/sub channel per job for WebSocket log streaming
  • PostgreSQL 16 — persistent storage for jobs, schedules, pipelines, reconciliation schedules, and saved configurations

Troubleshooting

Issue Cause Solution
“Configuration file not found” Missing local config Copy template from config/templates/ to config/local/ and edit paths
“No incidents found in batch mode” Extract files not in expected directory or not matching filename pattern Check batch.paths.extract_dir and filename_patterns.extract in config
UnicodeDecodeError on CSV read File uses non-UTF-8 encoding (e.g., Latin-1 from AS/400) The system auto-detects encoding (UTF-8 → UTF-8-sig → Latin-1). If issues persist, convert file: iconv -f ISO-8859-1 -t UTF-8 input.csv > output.csv
“ModuleNotFoundError: No module named ‘core’” Package not installed in editable mode Run pip install -e . from the project root
GUI fails to launch PySide6 not installed or display server unavailable Ensure pyside6 is installed: conda install pyside6
FIRDS/GLEIF cache is empty Cache not yet populated Run firds-refresh --type full or gleif-refresh --type full
Docker containers fail to start Docker not running or port conflict Ensure Docker Desktop is running. Check ports 80, 5432, 6379 are free. Run docker compose down then docker compose up --build
